## Main diagnosis

### What went wrong
1. **Routing / intent classification is weak for legal/compliance questions.**
   - L01 se fue a un intent de reporte de productos.
   - L11 cayó en una plantilla de ayuda sobre módulos del sistema.
2. **Transactional flows are only partially guided.**
   - C01 y C11 no ejecutan sin contexto, lo cual está bien.
   - Pero tampoco completan bien el slot filling ni hacen la confirmación robusta esperada.
3. **Reporting is functional but not fully shaped to spec.**
   - A01 omite desglose por método de pago.
   - A11 responde vacío correctamente, pero sin estructura consistente de "top productos".

### Likely root causes
- Falta de un clasificador/route explícito para **legal/compliance knowledge**.
- Falta de un flujo de recuperación RAG diferenciado entre:
  - ayuda de uso del sistema
  - conocimiento legal/normativo
- Entities incompletas en acciones transaccionales (`intentPayload: null` en descuento).
- Templates de respuesta no normalizados para casos "sin datos".

### Recommended fixes
1. **Agregar intent / router específico para legal-compliance**
   - Separar NOM, Distintivo H, COFEPRIS, higiene, etc. de reportes POS.
2. **Crear corpus RAG legal controlado**
   - Respuestas con disclaimer corto: "Esto es orientación operativa, no asesoría legal".
3. **Fortalecer slot filling en acciones**
   - Cancelación: mesa/orden + producto + motivo + confirmación.
   - Descuento: mesa/orden + porcentaje + motivo/autorización + confirmación.
4. **Normalizar formatos de reporte**
   - Si no hay ventas, mantener estructura esperada y marcar explícitamente cada sección vacía.
5. **Agregar tests de regresión para legal routing**
   - Cualquier pregunta con NOM, Distintivo H, COFEPRIS, higiene, manipulación → jamás debe caer en intents de ventas/reportes.
